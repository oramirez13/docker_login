Docker + Flask Project Summary Log out

From zero to a complete CRUD with Docker

This summary documents every step taken in this project, from installing Docker to this very login page working.

1. Verify and install Docker

Docker was confirmed to be already installed on Arch Linux, but the service was disabled and inactive. It was permanently enabled with sudo systemctl enable --now docker, which starts it and configures it to boot on every system restart.

2. Security decision: keep sudo

Running Docker without sudo failed due to permissions, since the user did not belong to the docker group. It was consciously decided to keep requiring sudo for every command, because belonging to that group is effectively equivalent to root privileges.

3. First container: hello world in Python

A basic Dockerfile was created with the instructions FROM, WORKDIR, COPY and CMD, fixing along the way an incomplete COPY and an invalid image name due to uppercase letters and spaces.

4. First API with Flask

A virtual environment was created, Flask was installed, and an API was written with routes / and /saludo, returning JSON with jsonify, first run outside Docker to understand Flask on its own.

5. Fix insecure Flask configuration

It was identified that debug=True together with host="0.0.0.0" exposes a dangerous interactive debugger on the local network. It was changed to debug=False, keeping host="0.0.0.0" because it is necessary for Docker.

6. Dockerize the API

requirements.txt was generated with pip freeze, and a Dockerfile was written that installs dependencies before copying the code, taking advantage of Docker's cache. Port 5000 was exposed and tested with docker run -p 5000:5000.

7. Database review

The concepts of table, row, column and primary key were reviewed, along with the four CRUD operations (Create, Read, Update, Delete) in SQL, highlighting the risk of omitting WHERE in UPDATE or DELETE.

8. Separate passwords from code with .env

It was explained why you should never hardcode passwords in code, as they would be exposed forever in Git history. A .env file was created with the MySQL credentials, protected with .gitignore.

9. MySQL and the API together with docker-compose

Two services were defined in docker-compose.yml: basedatos (MySQL) and api (Flask), using env_file for credentials, volumes for persistence, and depends_on for startup order.

10. Centralize configuration in config.py

Configuration logic was separated into its own file, with a Config class that reads environment variables once, following the pattern used in the industry.

11. Diagnose startup error with healthcheck

The API failed to connect because depends_on only waits for the MySQL container to start, not for it to be ready. A healthcheck was added with mysqladmin ping and condition: service_healthy, forcing the API to wait until MySQL was actually responding.

12. Complete CRUD: Create, Read, Update, Delete

The four routes were implemented: POST /usuarios to create, GET /usuarios to list, PUT /usuarios/id to edit, and DELETE /usuarios/id to delete. All using parameterized queries (%s) to prevent SQL injection, and checking existence before editing or deleting.

13. Gunicorn as production server

Flask's development server was replaced with Gunicorn, moving crear_tabla() outside the if __name__ == "__main__": block so it would also run when Gunicorn imports the application, instead of running it directly.

14. Login with hashed passwords

A password column was added to the table, using generate_password_hash to never store passwords in plain text, and a POST /login route that compares with check_password_hash, always returning the same generic error message to not reveal which emails exist.

15. Frontend with skeletor.html, Bootstrap and jQuery

A login form was built following the base template skeletor.html and the dark palette of custom CSS, connected to the API via Ajax with jQuery, enabling CORS on the backend with flask-cors to allow communication between different origins.

16. Diagnosis: Gunicorn logs

It was discovered that Gunicorn does not log requests by default, unlike Flask's development server. --access-logfile - was added to Gunicorn's command to see each request in real time, confirming the complete flow: form, OPTIONS request from CORS, POST, and successful response.

17. Real authentication with JWT

The simple sessionStorage flag was replaced with a JWT token signed by the server with PyJWT and a SECRET_KEY stored in .env. The /login route generates the token on authentication, and a @token_required decorator protects routes like /verificar-sesion, verifying the token signature on every request. This page now queries the server before displaying itself, instead of relying on a local flag alone.

18. Real logout: clear session on the client

Before, the "Back to login" link just redirected without clearing anything. If the user went back with the browser button, the dashboard was still visible because the token remained in sessionStorage. The link was replaced with a button that calls a logout() function which executes sessionStorage.clear() before redirecting, ensuring no residual session data remains. The window.location.replace() function is used instead of href because it doesn't leave the dashboard in the history, preventing the back button from showing it again.

19. Protect the user listing with a token

The GET /usuarios route was the only CRUD route that did not require authentication. Even without returning passwords, the list of emails and IDs is sensitive information that allows enumerating valid users. The @token_required decorator and token_data as the first parameter were added, making all CRUD routes require a token. This follows the principle of defense in depth: each security layer protects even if another one fails.

20. Configurable API URL in the frontend

Both script.js and dashboard.html had the server URL hardcoded as http://127.0.0.1:5000. It was extracted to a global API_URL variable at the start of each file, so that if the backend changes address, only one line needs editing. Specific routes (/login, /verificar-sesion) are concatenated to this variable. In real production, this variable comes from a build configuration file, not written directly in the code.

21. Clean up dead code and fix .gitignore

app.py was removed, which contained a single print() and was not used by the Dockerfile or any other file. The Dockerfile runs gunicorn ... api:app, which points to the app object inside api.py. .gitignore was also fixed: the virtual folder is called .venv/ (with a dot), but the file only excluded venv/ (without a dot), which meant the virtual environment would be uploaded to the repository.

Note: the token is stored in sessionStorage, which is still vulnerable to XSS attacks if an attacker manages to inject malicious JavaScript into the page. The more secure alternative, httpOnly cookies, requires more advanced configuration and is left as pending for a next level of learning.

Practice project: Docker, Flask, MySQL and frontend with Bootstrap.
